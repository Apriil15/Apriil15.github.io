The Clean Coder - 心得

Posted on Apr 25, 2021

前幾天把 The Clean Coder 看完了，記錄一下心得。大概說一下為什麼會買這本書，比較有想法的點，以及推不推薦。

為什麼會買

前幾週跟朋友去天瓏書局買書，拿了 Clean Code，以及幾本書。

結帳的時候，天瓏老闆就說，有另一本雙胞胎書籍，要不要順便拿，價格幾乎是買一送一。

友人 P 也說這本書是他第一本買完之後真的有看完的書。我翻了一下感覺還行，也沒想太多就順手結帳了 XD

事實上，我到現在還是不知道 Clean Code + The Clean Coder 這兩本書合起來是多少錢 XDD

比較有想法的點

書中有提到很多職場生態，例如：何時該說不、何時該說是、給出承諾代表的是什麼、Scrum、TDD，基本上就是大家的日常。

職業道德

職業發展是你自己的事，老闆沒有義務確保你能夠在職場立於不敗之地。如果把職業發展寄望老闆，結果將會很慘。你應該計畫每週工作 60 小時，前 40 小時給老闆，後 20 小時給自己。

前半段我是蠻認同的，其他多出來的我都當做 bonus。

20 小時的部分，近期是有的，而且應該還超出許多 XD

The Flow Zone

進入 zone 時，你可能可以寫更多 code，更快取得 TDD 的反饋。問題在於，進入 zone 時，你其實沒有顧及全域，因此，可能做出之後必須再重做的決策。

相較現在大家推崇的 zone，作者則有不同的看法。不過書中有指出，如果是自己在練習時，進入 zone 則是要追求的。

我自己是喜歡不受打擾，連對面同事會一直自言自語都覺得有點煩躁 XDD

TDD

作者看的出來非常推 TDD，也就是 Test-Driven Development。

這讓我一直反思目前工作上接手的幾個 legacy code，我遇到的狀況是這樣的：

不敢重構 -> 為什麼不敢重構 -> 怕改壞 -> 為什麼怕改壞 -> 沒有單元測試

以 TDD 作為開發模式是很有道理的！

團隊和專案，何者為先

團隊產生化學效應、默契建立是需要時間的，以專案為本，來建立團隊是愚蠢的。

概念上就是人員磨合需要時間，應該以團隊來分配專案，而不是以專案來組建團隊。

對於公司的組織上，有種淡淡的既視感啊 XDD

推不推薦

除了一些幾十年前電腦怎麼運作有點難體會，基本上這本書很好讀。一些真實案例也十分有趣，真的是日常上映中 XD

推薦，看就對惹！

後記

感覺平常在看就該把想法記下來..
